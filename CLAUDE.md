## Developer Preferences (Critical)

These are explicit working preferences. Follow them strictly.

**1. Virtual Environment First** — Before any Python or `uv` command, verify `.venv/` exists and is active. Never assume the environment is ready. Activate with `source .venv/bin/activate`.

**2. Confirm Before Making Changes** — Explain proposed changes first and ask "Should I proceed?" Do not create files or modify code without explicit approval.

**3. Clean Code** — PEP 8 (no mid-file imports, functions before classes). Use repository-relative paths only. Mark unverified assumptions explicitly with `, I assume`.

**4. No Bloat** — Do not add files, docs, wrappers, or abstractions unless they are: required for build/runtime/test/docs, reused at 2+ real call sites, remove duplicated logic, or measurably improve developer workflow. For any new artifact state its purpose, owner, and deletion criteria. Remove anything not referenced by runtime, build, or docs entry points.

**5. Explain Mode** — When giving implementation or architecture advice: explain the *why* behind design choices, include trade-off analysis against alternatives, and share a step-by-step plan before writing code.

**6. First-Principles Engineering** — Identify the underlying flaw and make the error state unrepresentable. Avoid shallow `try/except` patches and ad hoc fallback chains. Prefer root-cause fixes over local symptom treatment.

**7. Response Transparency** — End each response with a brief note stating what was uncertain, what was deferred, and what limitations affected execution.

**8. No Hardcoding** — Derive values from context before hardcoding. Order of preference: request/session data → application settings → computed/derived values → hardcoded constants (only when truly invariant, with explanatory comment). When an external API supports auto-detection, omit the parameter rather than defaulting to a hardcoded variant.

**9. Endogenous Fairness Only** — Do not introduce external fairness oracles, absolute fairness anchors, or prescriptive fairness gates. Fairness must be generated by the internal negotiation procedure itself.

## Architecture

The pipeline has three stages: **claim extraction → negotiation → fairness metrics**.

```
claims_extractor.py  →  negotiation_swarm.py  →  utils/metrics.py
```

### `negotiation_swarm.py` (primary module)

`NegotiationSwarm` drives a LangGraph state machine. Key design decisions:

- **Bounded role state**: role nodes see only `current_claim`, `last_accepted_claim`, `spectator_pov` — not full history — to prevent reinforcement cascades.
- **Full spectator state**: the impartial spectator reads complete history, emits `LOOP_STATUS` / `GRIDLOCK_SUMMARY` / `PROPOSED_POV`, and relays one neutral signal via `spectator_pov`.
- **Random routing**: actors selected randomly; the same actor cannot act twice in a row.
- **Version-based consensus**: any rewrite resets all roles to unsatisfied; all must re-confirm on the new version.
- **Termination**: all roles satisfied, or `max_iterations = len(roles) * 10`.

`NegotiationState` (TypedDict) carries: `current_claim`, `last_accepted_claim`, `proposed_claim`, `claims_object`, `adjustment_notes`, `spectator_reports`, `spectator_pov`, `satisfied_roles`, `role_last_confirmed_version`, `round_count`.

Return payload of `negotiate()` / `negotiate_async()`: `claims_object`, `adjustment_notes`, `spectator_reports`, `iterations`, `agreement_count`, `success`, `satisfied_roles`. `claims_object[0]` is the original (`version=0`); `claims_object[-1]` is the latest.

### `config.py`

Auto-detects cloud vs. local LLM from `USE_LOCAL_LLM` env var or `--local-llm` CLI flag. Exposes `get_api_key()`, `get_base_url()`, `get_model()`, `get_logger(name)`.

### `claims_extractor.py`

`decompose_claims(text)` extracts claims from contract text. `Claims.classify_claims()` asynchronously labels each as *factual* or *negotiable* via DSPy + RAGAS.

### `utils/metrics.py`

Cross-encoder semantic distance utilities for measuring drift between the initial and final claim.

## Fairness Metrics

- **Dispute Score** = `total_rounds / num_negotiable_claims` — higher means more initial injustice.
- **Semantic Distance** = cross-encoder distance between initial and final claim — higher means greater semantic drift.
